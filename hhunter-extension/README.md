# HHunter — расширение Chrome (Manifest V3)

Отклики на [hh.ru](https://hh.ru) через ваш бэкенд HHunter: сервер генерирует сопроводительное письмо (Groq), расширение подставляет текст в форму на hh.ru и сохраняет результат через API.

## Требования

- Запущенный бэкенд HHunter (FastAPI) и вход в веб-интерфейс на том же origin, что указан в popup.
- В **Настройках**: ключ Groq, модель и текст резюме; в **Параметрах поиска** сохранены текст запроса и фильтры (они уходят в `GET /extension/settings` для сборки URL поиска на hh.ru).

## Установка

1. Откройте `chrome://extensions`, включите **Режим разработчика**.
2. **Загрузить распакованное расширение** → выберите папку `hhunter-extension` из корня репозитория.
3. В popup при необходимости укажите:
   - **URL API** — тот же хост, что в `VITE_API_BASE` фронта (обычно `http://localhost:8000`).
   - **URL сайта HHunter** — origin фронта (обычно `http://localhost:5173`), откуда читается `localStorage` с `hhunter_token`.
4. В `manifest.json` в `host_permissions` должны быть ваши origin фронта и API (для production добавьте строки с `https://…`).

## Режимы работы

### Активная вкладка (по умолчанию)

1. Войдите на сайт HHunter, сохраните JWT в popup (**Сохранить**).
2. Откройте одну вакансию `hh.ru/vacancy/{id}`.
3. **Запустить** — один прогон по текущей вкладке: письмо вставляется в форму (`setNativeValue` + `input`/`change` для React), на сервер уходит статус `sent` без обязательного нажатия «Отправить» на hh.ru (отправку лучше проверить вручную). Цикл сам останавливается, чтобы не дублировать отклик на той же странице.

Ответ `GET /extension/settings`: параметры поиска лежат во вложенном объекте **`search`** (как на сайте HHunter), а не в корне JSON — это ожидаемо для `buildSearchUrl`.

### Полный цикл (галочка в popup)

1. Включите **«Полный цикл: поиск на hh.ru → вакансии → отклик»**.
2. **Запустить** — в фоне открывается выдача поиска по параметрам из сайта HHunter, собираются id вакансий (скролл SERP), по очереди открываются карточки, генерируется письмо и **автоматически нажимается отправка**; в БД `sent` только если удалось подтвердить успех в DOM.
3. Используются дневной лимит и паузы из раздела «Поиск»; перед генерацией запрос `vacancy-known` пропускает уже учтённые вакансии.

Остановка цикла — **Остановить** в popup. При обнаружении капчи расширение **останавливает полный цикл** автоматически (с сообщением в статусе); дальше — вручную. При **429** (дневной лимит на сервере) полный цикл тоже останавливается.

## Ошибка «message channel closed» / «asynchronous response…»

Chrome ругается, если вкладка **закрыла канал** до вызова `sendResponse` в content script: перезагрузка страницы, закрытие вкладки, зависание, или ответ позже таймаута. Сделайте **F5** на странице вакансии и повторите. Фон ждёт ответ до **120 с**, затем закрывает вкладку вакансии в полном цикле и пишет ошибку в логи.

## Ограничения и риски

- Массовые DOM-отклики должны **соответствовать правилам hh.ru**; соблюдайте паузы и лимиты, иначе возможны блокировки или капча.
- Вёрстка и `data-qa` на hh.ru меняются — при сбоях проверяйте селекторы в `content.js` / инъекции SERP в `background.js`.
- Лимит откликов на сервере считается по **UTC** (как в API `save-application`).

## Структура

- `manifest.json` — разрешения и точки входа.
- `background.js` — service worker: режимы `active_tab` / `full_auto`, цикл SERP и `chrome.scripting`.
- `content.js` — страницы hh.ru: генерация, `setNativeValue`, отправка формы, `save-application`.
- `popup/` — интерфейс, токен, выбор режима.
- `utils/api.js` — `fetch` к бэкенду.
- `utils/searchUrl.js` — сборка URL `https://hh.ru/search/vacancy?...` из объекта `search`.

Пошаговый журнал уходит на сервер (`POST /extension/log`) и отображается на сайте HHunter во вкладке **«Логи»** (опрос раз в ~1.5 с). Сообщение в консоли про «message channel closed» см. в README выше — обычно F5 на вкладке вакансии или таймаут ответа content script.
